在 Raspberry Pi 4B 上安装 Fedora CoreOS

Jan 15, 2026

摘自 Fedora CoreOS Documentation :

Fedora CoreOS is an automatically updating, minimal, monolithic, container-focused operating system, designed for clusters but also operable standalone, optimized for Kubernetes but also great without it. Its goal is to provide the best container host to run containerized workloads securely and at scale.

简而言之，最小化维护，不可变，极致稳定，容器中心。

总体流程

Fedora CoreOS 通过操作系统镜像的方式分发安装介质和系统更新；从第 43 版开始，Fedora CoreOS 默认使用 bootc OCI 镜像分发。这意味着每份镜像都是完全相同的（包括/etc里的配置文件）。为了实现操作系统配置的自定义， Fedora CoreOS 使用一个类似于 cloud-init 的 ignition 机制来实现系统的初始配置。

另外，我们的目标平台，树莓派 4B，与传统 PC 平台又有所不同。

于是，我们的安装步骤变成了：

  1. 获得 ignition 配置文件
  2. 安装 Fedora CoreOS
  3. 针对树莓派用 GPU 启动 CPU 的神必启动方式进行特殊处理

3. 对树莓派做特殊处理

最魔幻的一个阶段来了。

这里说明一下本人猜测的 Raspberry Pi 4B 的大致启动流程：

  1. VideoCore IV GPU 启动，加载固化在 GPU 里的一阶段启动引导器；
  2. 一阶段启动引导器读取板载 EEPROM 里的二阶段启动引导器；
  3. 二阶段启动引导器读取 EEPROM 里的配置信息，按照指定的启动顺序尝试启动；
  4. 从本地存储介质启动时，要求介质第一个分区格式为 FAT32，根目录下需要有 start.elf VideoCore 可执行文件；
  5. 加载 start.elf 程序，启动第三阶段启动引导器；
  6. start.elf 会寻找第一主分区根目录下的 config.txt 并按照这个配置文件设置外围设备（如 GPIO），加载内核、initramfs 以及 devicetree 和 devicetree overlay；
  7. 最终，启动 bcm2711 ARM CPU 核心，将控制权转交内核。

而具体到 Fedora CoreOS，情况又变得稍微有那么亿点点不一样。为了维持一致性，整个 Fedora 发行版系列都采用 GNU GRUB 作为启动引导器。而为了在类似于树莓派这样的“嵌入式” ARM 设备上启动（并维持一致性），Fedora 又给 GRUB 套了一层 U-Boot。于是，我们现在有了 5 层启动引导器。

禁止套娃

我们需要确保第二阶段启动引导能够找到第一主分区以及其中的 start.elf。而在我的安装过程中，现在的情况是：

  1. 第一分区是一个类型为 Linux reserved 的无格式分区（大概是为了留出空间给 GRUB 嵌入 grub.img）；
  2. 第二分区是一个 FAT16 格式的 EFI System Partition，里面存放的是 grub.efi 以及其他启动引导相关的 EFI 程序；但由于这是一个通用系统映像，它假设你的系统是一个 ARM SystemReady 平台（说人话就是 EFI 平台），并没有树莓派相关的启动固件。

于是我们需要：

  1. 调整分区表让第二阶段启动引导器能启动第三阶段启动引导器；
  2. 重新格式化 ESP 为 FAT32；
  3. 获取第三阶段启动引导器以及 U-Boot。

Fedora CoreOS 食用方法

系统更新

这玩意是自动更新的，所以大概率不需要你手动管它。

如果你想手动更新，可以使用 zincati-update-now 命令来让 zincati 检查一次更新。

注意，截至文章写作时间(2026/1/15，CoreOS 版本 43.20251214.3.0)，此命令尚处于试验阶段，且在将来的更新中可能会有更改。

如果你熟悉红帽子那套 rpm-ostree 包管理器，而且尝试用它来发起手动更新，你应该会注意到 rpm-ostree 说更新由zincati 管理，不能直接用 rpm-ostree 来更新。这时，你可以采用一个比较 hacky 的方法，就是停掉 zincati.service 来使 rpm-ostree 进入没有 update-driver 的手动管理状态。

系统回滚

如果系统滚挂了，你可以在启动时把键盘连接到树莓派，然后在 GRUB 启动选单里选择上一个 OSTree Deployment 启动。启动到上一个 deployment 之后，记得运行 rpm-ostree rollback 来将回滚的 deployment 设为默认启动项，并且停掉 zincati.service 以免又自动更新到新版本。

运行应用程序/服务

Fedora CoreOS 是“容器中心”的，也就是说你应该以容器的形式运行大部分应用程序。关于容器的使用方法，我目前还没有写相关的文章，请自行上网搜寻。

系统内预装了 podman 和 docker，可以开箱即用地拉取、构建容器镜像，以及运行容器。要让 core 用户的容器在 core 用户登出后继续运行，可以使用 loginctl enable-linger 来让 core 用户的 systemd 用户服务管理器在用户登出后继续运行（也会在系统启动时自动启动）。

注意，如果你的应用程序需要 root 特权，比如直接控制 GPIO 之类的，你需要以 root 身份拉取、构建镜像并运行容器。

podman 没有一个中心化的守护进程，如果想配置容器开机启动需要借助 systemd。可以手搓 systemd 单元文件（我就是），也可以用 podman 提供的一个 systemd generator，参见podman-systemd.unit(5) 手册页。

安装命令行工具

Fedora CoreOS 预装了 toolbx(或者你也可以叫它 toolbox)，这是一个利用容器技术，在不可遍发行版上提供类似于传统发行版体验的工具。在保持根文件系统隔离的同时，与主机共享用户主目录、ipc, net, pid 等命名空间， /dev设备节点，以及dbus, X11 与 Wayland 套接字的互通。

BTW，隔壁同为 Fedora 旗下的 Fedora Atomic Desktops 系列发行版也自带这个工具。

摘自 https://containertoolbx.org：

Toolbx is a tool for Linux, which allows the use of interactive command line environments for software development and troubleshooting the host operating system, without having to install software on the host. It is built on top of Podman and other standard container technologies from OCI.

Toolbx environments have seamless access to the user’s home directory, the Wayland and X11 sockets, networking (including Avahi and CA certificates), removable devices (like USB sticks), systemd journal, SSH agent, D-Bus, ulimits, /dev and the udev database, etc.. The host file system can be accessed at /run/host.
